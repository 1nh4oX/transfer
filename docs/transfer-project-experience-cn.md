# Transfer CloudSpace 项目经历（中文）

## 一、项目概述
我独立完成了一个全栈云文件管理系统 **Transfer CloudSpace**，目标是将“文件存储、共享分发、轻量笔记”整合为一个可在公网访问的跨平台应用。项目覆盖了从需求定义、接口设计、功能实现到部署上线、故障排查和流程固化的完整软件工程闭环。

- 项目周期：2026 年 3 月
- 技术栈：Go、Gin、PostgreSQL、React、Vite、Nginx、systemd、Docker（数据库）
- 部署环境：阿里云 ECS（Ubuntu 22.04）

## 二、核心功能与系统设计
### 1. 用户与鉴权
- 实现登录/注册接口：`POST /api/auth/login`、`POST /api/auth/register`
- 使用 JWT 进行会话认证，受保护路由统一通过中间件校验
- 使用 bcrypt 进行密码哈希存储，避免明文口令落库

### 2. 文件与目录管理
- 实现文件上传、列表、下载、删除、移动等完整文件生命周期能力
- 实现目录树、新建文件夹、目录移动，支持层级组织
- 文件接口支持分页，兼顾性能与可扩展性
- 下载使用后端文件流式返回，保证大文件基础可用性

### 3. 分享与公开下载
- 实现分享链接创建：`POST /api/shares`
- 实现公开分享信息查看：`GET /api/s/{token}`
- 实现公开下载：`GET /api/s/{token}/download`
- 分享 token 采用随机 URL-safe 生成方案，并支持过期时间校验

### 4. 速记 notes 功能
- 实现个人 notes 的创建与分页查询：`POST /api/notes`、`GET /api/notes`
- 前端提供右侧速记区，支持快速输入、回车发送与自动滚动

### 5. 架构与工程组织
- 采用 API-first 思路（先定义 OpenAPI，再推进实现）
- 后端采用分层架构：`handler / service / repo / model`
- 错误处理规范化：统一错误码、统一响应结构，提升前后端联调效率

## 三、前端工程与体验优化
前端基于 React + Vite 构建，除功能接入外重点解决真实场景中的可用性问题：

- 登录/注册错误提示与 Toast 反馈机制
- 登录失效（401）自动清理 token，降低异常状态残留
- 批量上传结果反馈（成功/失败分开提示）
- 分享链接复制容错：Clipboard API 失败自动降级复制
- Office 预览增加超时与失败兜底，避免长时间无反馈
- 修复 notes 区域滚动与高度链路问题（`h-screen/min-h-0/overflow-y-auto`）
- 增加空间占用显示与 2GB 配额前端拦截逻辑

## 四、部署上线与故障治理
### 1. 部署方案演进
项目早期采用“服务器在线构建 + Docker”方式，在低规格 ECS 上稳定性较差。后续收敛为：

1. 本地交叉编译 Linux amd64 后端二进制
2. 使用 `scp` 上传服务器
3. 通过 `systemd` 托管后端服务
4. 使用 Nginx 统一处理 `/api` 反代与前端静态托管

该方案显著提升了发布稳定性与可重复性。

### 2. 关键问题与解决
- **构建与网络问题**：Docker 拉取超时、Go 依赖下载超时
  - 处理：镜像/代理优化，减少服务器重负载构建
- **资源问题**：小内存 ECS 导致构建卡顿与 SSH 不稳定
  - 处理：增加 swap，并迁移到本地构建发布
- **运行问题**：端口冲突（80/8080）、Nginx default_server 冲突
  - 处理：清理旧进程/旧容器，统一入口配置
- **联调问题**：401/404 交替出现、请求命中错误目标服务
  - 处理：固定前端 API 基址，统一反代路径
- **上传问题**：Nginx 报 `413 Request Entity Too Large`
  - 处理：配置 `client_max_body_size`
